Non-Blocking Message Delivery Architecture

Michael Hayes · LaunchMate

Building autonomous pokes meant solving a constraint: agents can't wait around for message delivery to complete. The solution was obvious. The interesting part was choosing the simple version.

1/4

The Starting Point

I'm building the execution side of Agent Pokes — cron jobs and agent-triggered workflows that autonomously deliver messages to founder agents.

The pattern is simple:

  • Founder schedules a poke ("generate weekly tidbit every Monday 9am")
  • Cron triggers → agent executes multi-step workflow
  • Agent delivers result message to founder

But there's a constraint: the sender agent can't be blocked waiting for message delivery. Letta's send_message_to_agent waits for a response — that could take 10-30 seconds if the founder agent is processing something complex.

My question: "Should I just queue messages on my server?"

2/4

Three Infrastructure Approaches

Database Queue
✓ Zero new infra
✓ Ship today
✓ Native scheduling
✗ Polling overhead
Redis + Worker
✓ Instant delivery
✓ Built-in retries
✓ Production-grade
✗ New dependency
Vercel Background
✓ Built into Vercel
✓ No extra infra
✗ 5min limit
✗ Limited retries

Redis is objectively "better" — instant processing, proven at scale, built-in retry logic. But better for what use case?

3/4

The Decision: Database Queue

I chose Option 1: Supabase table + cron worker.

CREATE TABLE message_queue (
  id UUID PRIMARY KEY,
  recipient_agent_id TEXT,
  message_content TEXT,
  scheduled_for TIMESTAMP,
  status TEXT, -- pending/delivered/failed
  retry_count INT DEFAULT 0
)

Why not Redis? Because I already have Supabase, and the constraint isn't latency — it's "don't block the sender." A 30-second polling delay doesn't matter when the alternative is 2 days of Redis setup + deployment config.

This maps to the reliability-first decision from Feb 16: ship dependable infrastructure with what you have, optimize when you hit the constraint.

Current status: Architecture decided. Next: worker implementation + retry logic + execution log storage.

Upgrade trigger: If we hit >1000 messages/day or need sub-second delivery, migrate to Redis. Until then, "good enough" ships today.
