The pattern

Implement first, think later

Each auth pivot cost hours of implementation and debugging. The pattern: hit a wall → try a new approach in code → discover a new wall → repeat.

The core mistake wasn't any individual decision. It was implementing before the auth model was settled. Three different auth approaches were built and abandoned before anyone asked: "What's the simplest thing that actually works?"

The answer was sitting in the codebase the whole time. Clerk already handles Google OAuth. MCP clients already have Google tokens. Just validate the token and resolve the user by email. No DCR. No PKCE. No custom OAuth tables. No pepper.

The Clerk detour was the most expensive dead end. Clerk manages browser sessions via cookies. MCP clients need bearer tokens. These are fundamentally different auth models. A 10-minute architecture conversation would have caught this. Instead: bridge routes, custom middleware, and days of debugging before the mismatch became obvious.

The v1.1 spec

Simple, scoped, server-to-server

The final architecture isn't just simpler. It's a different product than what we started building.

What we started with

Founders connect Cursor directly to LaunchMate. Generic message passing. Custom OAuth stack. Founder-facing.

What v1.1 actually is

Poke orchestrates between Granola and LaunchMate. Transcript ingestion. Google passthrough auth. Server-to-server.

Five tools, one auth path, one orchestrator.

The tool surface tells the story: this is orchestration infrastructure, not a chat interface. Poke resolves who to talk to. Poke pushes the transcript. The founder never opens MCP directly — their tools do it for them.

The takeaway

The auth was never the problem

The problem was building before deciding. I kept implementing my way out of uncertainty instead of stopping to think. Each pivot felt like progress because I was writing code. But code without a settled design is just expensive exploration.

The three rules I'm taking away:

  • Decide auth before writing auth code. "How does the client authenticate?" is a design question, not an implementation question. Answer it with a conversation, not a PR.
  • Check what you already have. The solution (Google OAuth via Clerk) was in the codebase the whole time. I built three new systems before looking at the one that already worked.
  • Narrow the scope, name the client. "External tools can connect" is too vague. "Poke orchestrates transcript ingestion" is buildable. Specificity isn't a constraint — it's a gift.

The MCP server works now. But the three weeks it took could have been one. The delta wasn't technical skill — it was the discipline to stop coding and start thinking.
