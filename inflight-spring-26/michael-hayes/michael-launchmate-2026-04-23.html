What I got wrong

Two times I tried to skip the hard thing

The v1.1 Google passthrough was seductive. "We already use Clerk. Just validate Google tokens and resolve by email. No custom OAuth, no DCR, no token tables."

It works — if the only MCP client is Poke. Poke is a backend service. It already has a Google token. It just needs to prove who it is.

But then Cursor shows up. Cursor is a public client running on a founder's laptop. It can't store client secrets. It needs to dynamically register itself. It needs PKCE. It needs an authorization flow that redirects through a browser. Google passthrough doesn't give you any of that.

The simplification wasn't wrong — it was incomplete. It solved for one client and assumed the others didn't matter.

The deeper mistake was confusing "simpler" with "sufficient." Google passthrough is simpler. But it only handles the server-to-server case. If Cursor is a real MCP client — and the connector thesis depends on it being one — then DCR and custom OAuth aren't over-engineering. They're the minimum viable auth.

What actually shipped

OAuth 2.0 Authorization Server with DCR

The final design addresses every gap that came before:

Previous gaps
  • No scope enforcement
  • No agent ownership checks
  • No token revocation
  • No PKCE for public clients
  • Tokens stored raw in DB
  • No redirect URI validation
Final design
  • read / write scope per tool
  • User-owned agent checks on every call
  • /revoke endpoint
  • PKCE S256 required for public clients
  • Peppered SHA-256 hashing
  • Registration + allowlist rules

The auth flow: MCP client discovers via .well-known → registers at /register (DCR) → user authorizes at /authorize (Clerk backs the human login) → client exchanges code at /token → bearer tokens validated on every MCP request → scope + ownership enforced per tool.

Clerk still handles identity. LaunchMate OAuth handles authorization. That separation is clean — Clerk proves who you are, the OAuth server proves what you're allowed to do.

What I actually learned from three weeks of auth pivots:

  • Name every client before writing auth code. "External tools" is too vague. "Cursor, Claude, and Poke" is buildable. Each has different auth requirements.
  • "Simpler" ≠ "sufficient." Google passthrough was simpler but only handled one of three clients. The right architecture handles all of them.
  • Don't escape hard problems — scope them. I kept trying to avoid DCR instead of asking: "Which clients actually need DCR, and what do they need from it?"
  • The solution was the hard thing all along. I built three shortcuts before building the thing that actually works. The delta wasn't technical skill — it was the discipline to sit with complexity long enough to understand it.
