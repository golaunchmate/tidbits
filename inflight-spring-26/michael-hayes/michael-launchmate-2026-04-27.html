The Maintenance Ping

How do you track what someone's working on without interrupting them? The answer turned out to be: don't track. Just notice.

The connector thesis meets Cursor. The question isn't "how do we get data?" — it's "how do we get data without being the thing the founder closes?"

The tension

The connector thesis has a last-mile problem

The pivot was clear: founders shouldn't type updates into LaunchMate. Their tools should push context automatically. Cursor edits code. Granola records meetings. Poke orchestrates. LaunchMate just... receives.

But "automatically" hides a design question that took a week to answer: how does LaunchMate know what the founder did in Cursor without becoming the thing that slows Cursor down?

The obvious answer — "run a background process every 30 minutes" — doesn't work in Cursor. Cursor isn't a server. It's an editor. Background processes in editors are intrusive, unreliable, and they eat tokens the founder is paying for.

The real constraint: LaunchMate should learn about progress the way a good teammate does — by noticing what's happening, not by interrupting to ask "what are you working on?"

Three approaches that felt right but weren't:

  • Hook calls MCP directly. But the MCP session belongs to the active Cursor client. A separate API key would reintroduce the auth problems we just spent three weeks solving.
  • Hook spawns a background Cursor agent. Recursive agent loops. Hidden token spend. Permissions confusion. The founder doesn't know what's running on their machine.
  • Hook summarizes and sends. Summarization takes seconds. The hook blocks the editor while it thinks. Every keystroke has a latency tax.
2/4
The pattern

Separate noticing from processing

The solution was to split the problem in two: a hook that notices, and a skill that processes.

Hook (event collector)

Runs on every file edit, shell command, or session start. Does one thing: writes a timestamp to a local marker file. Takes <1ms. Never blocks. Never calls the network. Never talks to LaunchMate.

If it crashes, the founder never notices.

Skill (the processor)

Runs when the founder next talks to their Cursor agent. Checks the marker. If enough time has passed, summarizes recent work, routes it to the right LaunchMate agent, and creates drops for follow-ups.

Only runs when the founder is already interacting — no background token spend.

We're calling it a "maintenance ping" instead of a "cron job" because the language matters. A cron runs on a schedule regardless of activity. A maintenance ping only fires when there's activity to notice. Activity-throttled, not time-throttled.

The MVP pattern: sessionStart hook + activity-throttled afterFileEdit / afterShellExecution. The hook writes the marker. The skill reads it. The founder never waits.

3/4
The implication

Progress tracking that respects attention

The maintenance ping is the connector thesis made real for Cursor. The founder works. The hook notices. The skill processes. LaunchMate learns. Zero interruption. Zero manual input. Zero background tax.

What it doesn't solve — yet:

  • The 3-day gap. If the founder doesn't talk to Cursor for three days, the marker sits unread. The ping only processes when there's an active conversation.
  • Cross-app tracking. This works for Cursor. Notion, Google Drive, and other tools need their own integrations. The long-term answer is a local helper or a server-side progress endpoint — but that's not the MVP.
  • The due marker mechanism. The hook still needs a concrete way to tell the active Cursor agent "hey, run the progress skill." A local file marker is the leading candidate, but the exact implementation is TBD.

But the architectural principle is locked: separate noticing from processing, and only process when the founder is already engaged. Everything else is implementation.

The connector thesis says founders shouldn't have to update LaunchMate. The maintenance ping is how we deliver on that promise for the tool they use most — without becoming the thing they close.
