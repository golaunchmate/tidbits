What happened

A newsletter item changed everything

Craig subscribes to a Google Labs service called CC — an AI that reads his email and gives him a morning briefing. One morning, it said: "You might want to sign up Delilah and Brody for this new after-school program from San Diego Unified School District."

Delilah and Brody are his girlfriend's kids. He doesn't get emails from the school district. He's never told CC about them.

What CC did: it read a San Diego news newsletter, found one item buried inside it about an after-school program, and connected it to something it knew about Craig's personal life. One item in a newsletter of seventeen. Relevant because of memory, not because of keywords.

"It took one entry in a newsletter and said, would this be useful for Craig? Oh, he's got kids. That's amazing."

Craig's response: "If LaunchMate could do that — surface something I didn't know I needed because it knows me — I'm all in."

That's the product. Not the pipeline. Not the chat interface. The moment where context becomes magic.

2/5
What we're building

Transcript in, drops out

Craig and Michael whiteboarded the full pipeline. The architecture finally has a concrete use case:

Conference call
→
Granola transcript
→
Transcript agent
→
Drop agent
→
Scheduled drops
→
Asset delivered

The key architecture decisions:

  • Drop agent is separate from transcript agent. Transcript agent analyzes. Drop agent routes and schedules. Separation makes both more generalizable.
  • Drops go to the best candidate agent — not always the founder's primary. A specialized agent might be better for certain tasks.
  • System messages replace agent-to-agent chat. One-way, reliable, no recursion risk. The transcript agent sends its analysis to the drop agent via system prompt.
  • Poke is the notification channel. Don't build LaunchMate's own notification system. Lean on Poke for iMessage delivery.
3/5
The shift

LaunchMate becomes a backend

The scrum crystallized something that's been emerging for weeks: LaunchMate's chat interface isn't the product. It's the admin panel.

Craig: "I haven't logged into Poke in a month. But I'm reading all the emails of all the things it's doing."

The reframe:

Old ModelNew Model
Founder logs into LaunchMateFounder gets a Poke message
Chat is the primary interfaceChat is the power-user interface
Agent waits to be askedAgent proactively does work
Drops are notificationsDrops are artifacts with context
Founder must check the appDeliverables come to the founder

The implication: the most important feature isn't what founders see in LaunchMate. It's what shows up in their Poke messages with enough context to act on immediately.

4/5
The exit criterion

One pipeline before I leave

Craig named the goal: "Before you leave, we successfully transition LaunchMate to this new model of context ingestion and proactive worker on your behalf."

That means one concrete thing: a transcript goes in, a drop comes out, and a founder gets notified. Once. End-to-end. That's the proof.

What's still open:

  • Asset storage. GitHub repo like tidbits, but generalized. Plus a viewing UI so founders can see their drop history.
  • Agent routing. Who does the drop go to? The primary agent? A specialized agent? The drop agent needs a directory.
  • Agent mail. Every founder gets an email address their agent can use. Baseline capability for outbound drops.
  • Drop context. The notification needs to explain why the drop exists — "last Friday you talked about this, so I..."
The CC moment isn't about reading email. It's about connecting something the person wouldn't have connected themselves. That's the bar. If a drop can do that even once, the whole architecture is validated.
