How we store tokens

And the single point of failure I don't love

This is the first time LaunchMate handles logging in on behalf of multiple users. Every connector token — Granola, Notion, Google Drive — gets stored per-user in Supabase.

But not in plain text. Here's the model:

Supabase DB └── integrations table └── ciphertext (AES-256 encrypted token) │ │ encrypted with key from ↓ │ Vercel (environment variable) └── ENCRYPTION_KEY

Safe scenario: Someone gets the database but not Vercel → tokens are mathematically irreversible. They're safe.

Risk scenario: Someone gets Vercel → they also have database access (it's our infra). Everything's compromised. That's the single point of failure.

Upgrade path: Amazon KMS — keys aren't in your code at all, you fetch them at runtime. Not implemented yet, but Laura should know it's the next step.

Also: dev/prod Clerk instances share one Supabase DB. Deduplication handles coexistence.

Two ways to connect

Not all connectors are built the same

Granola (direct MCP) Give it an MCP link → user signs in via OAuth → token stored in encrypted table → auth happens at web app level Simple. Just works. Notion / Google Drive (OAuth client) You manage an OAuth client in Google Console → custom consent screens ("Launchmate wants to access your Notion") → API-wrapped implementation → same encrypted token storage More complex. More moving parts.

I walked Laura through scopes using Discord as the example: your account has name, email, friends list, message data, servers, credit card, 2FA. A scope defines the green zone — what the application is allowed to see. Everything else is invisible.

Same pattern as the Discord bot: create a client, set what you want to access, consent screen shows the green zone.

Laura needs a non-school email to get added to Google Console.

Why we didn't use Poke

The intermediary that became a liability

My original assumption was simple: Granola → Poke → LaunchMate. Poke would sit in the middle, pull transcripts from Granola, and push them into the LaunchMate agent.

What I realized: this required building a LaunchMate MCP server into Poke, loading tools into Poke's context window, and threading messages from Poke into the LaunchMate agent. Poke became a true middleman — you can't just pass data through for free.

And then there was the threading issue:

"I'd be having a conversation and out of nowhere it'd stop responding. Then I'd say 'any update?' and it'd reply 'Hey there, can I help you?' — felt like messages were going into different context windows."

The principle I landed on:

  • Poke = notifier → if it fails, a notification doesn't arrive. Acceptable.
  • Poke = data manager → if it fails, meeting syncing breaks. Unacceptable.

Direct connector reduces failure points. If the team wants Poke to handle some of this later, it's possible — just a design choice, not a technical limitation.

Maps to the reliability-first pattern from February: ship what you can depend on.
4/5

What I couldn't remember

The cognitive debt I keep running into

Halfway through explaining the encryption model to Laura, I realized I couldn't recall the exact implementation. I knew I'd used AES-256. I knew it was safe. I'd verified it with Claude. But the details — the exact format, the edge cases — were gone from my head.

I had to pull up my Claude chat history to verify what I'd built.

What I said to Laura: "This is something I don't like about using AI — a lot of the stuff I do just goes away and you can't remember it."

This is the same cognitive debt concern I've been tracking. AI helps you ship, but the knowledge doesn't always stick. I built encryption I can't fully explain from memory. That's a real risk for handoff — Laura needs docs, not just demos.

Also discussed: Agent tool bloat (recommendation: max 8 tools). Explored a bash sandbox concept — agent writes bash that calls the LaunchMate API instead of loading connector tools into the context window. Keeps the window clean while preserving capability.
