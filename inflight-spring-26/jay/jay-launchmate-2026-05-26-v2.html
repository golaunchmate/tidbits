Technical Foundation

How Agents Actually Work

You need to understand this because your feature is literally a tool call — the mechanism described below.

User sends message →
Model: "Can I answer this myself?" → No
Model: "Any tools available?" → Yes
Model selects tool + parameters →
Emits tool call message (JSON) →
Python intercepts, parses, executes real code →
Composes tool call result message →
Sends back to model →
Model continues or stops
The key insight

The bridge between LLM land and code land is a message.

  • LLM → code: inspect tool_calls on the response object
  • Code → LLM: compose a tool call result message
Agent vs. Chatbot

An agent decides what it does AND when it stops. A chatbot is single-turn. LaunchMate uses agents. Your "set primary agent" tool will be one of the tools the agent can call to do real work.

Your First Task

Set Primary Agent via Chat

🎯 The Feature

Right now, founders can change their primary agent through the profile page on the web app. But there's no way to do it from the chat interface — meaning founders have to leave the conversation to make the switch.

You're building a tool call that lets founders say "make my transcript agent the primary one" right in the chat, and have it update the database without leaving the conversation.

Here's what you need to know:

Current architecture
  • Each user can have multiple agents (founder agent, transcript agent, etc.)
  • Agents are linked to users in Supabase via the agents table
  • The "primary" agent is determined by a field in the database (likely is_primary or ordering)
  • The profile page already has UI to switch which agent is primary
  • Agent IDs are UUIDs; the web app matches users by school email
What your tool call needs to do
  • Accept an agent_id parameter
  • Verify the agent belongs to the requesting user (ownership check)
  • Update the primary flag in Supabase — same logic the profile page uses
  • Return confirmation to the agent so it can tell the founder it worked

⚠ Watch the agent-to-agent looping problem: model updates (4.5→4.6) once caused agents to talk back and forth endlessly, draining $100-200 overnight. Your tool should be a simple write, not a conversation trigger.

Reliability > Features

First cohort rejected the app because it was buggy. Second cohort adopted it because it felt solid. Trust through reliability is the foundation for everything. Ship it working, not fast.

Dogfood everything

The team uses what it builds. Your feature should work for you too — if you can't see yourself using it, rethink it.
