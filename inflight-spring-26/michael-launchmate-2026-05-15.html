What we started with

Problem + the connector thesis

I built LaunchMate to be a memory-first assistant for ZIP founders. The current pivot is explicit: not another chat interface, but a connector-backed memory hub that owns context and creates artifacts founders can act on.

Starting assumptions that led here:

  • Founders don't type updates — connectors should surface work from their tools.
  • Memory ownership is defensible; LLMs will remain commoditized.
  • Meaning matters: effort visibility (artifacts + execution logs) is how founders perceive value.

What landed (and why)

Pivots, trade-offs, and provider constraints

We iterated the MCP auth model (multiple pivots → final OAuth 2.0 DCR with PKCE and peppered tokens), connected Granola/Notion/Google Drive, and rebranded Pokes → Drops to make work visible.

Trade-offs: Ship reliable connectors now (manual sync, dedupe, demo) vs fully automated pipelines later (rate limits, provider plans, backoff).

  • Granola rate limits forced manual sync → cron disabled to avoid noisy imports. Free plan only gives summaries, not transcripts. Michael
  • Discord Remote showed pre-training works, but inbound flows broke on a critical Tanner request — reliability matters more than novelty.
  • MCP integrations need push/webhook standards — the current pull model is frustrating. Craig
  • Drops replace raw summaries: context only matters when converted into artifacts + visible effort.
  • If providers limit us, we can build it ourselves — the barrier to self-host transcription (Parakeet/Quinn) is low. Michael
Design signal: make the value visible (effort logs / checkmarks).
